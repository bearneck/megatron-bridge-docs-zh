# 容错性

Megatron Bridge 集成了来自 [NVIDIA 容错扩展](https://github.com/NVIDIA/nvidia-resiliency-ext) 的弹性训练功能。该扩展提供了容错能力，有助于最大限度地减少因训练期间的故障和中断而导致的停机时间。

本页是稳定概述，说明每个容错功能是什么、何时使用以及哪些约束是持久的。关于操作设置、配置选项、参数表、代码锚点和验证命令，请参阅 [skills/resiliency/SKILL.md](../skills/resiliency/SKILL.md)。

## 功能概述

| 功能 | 目的 | 成熟度 | 集群 |
|---|---|---|---|
| 容错 | 挂起检测 + 自动作业重启 | 生产就绪 | 仅限 Slurm |
| NVRx 掉队检测 | 识别慢速 GPU | 生产就绪 | 任意 |
| 抢占处理 | 在时间限制前优雅关闭 | 生产就绪 | 仅限 Slurm |
| 异步检查点保存 | 非阻塞的检查点写入 | 生产就绪 | 任意 |
| 本地检查点 | 快速本地保存与复制 | 生产就绪 | 任意 |
| 重运行状态机 | NaN / 尖峰损失归因 | 实验性 | 任意 |
| 进程内重启 | 在同一进程内重启 | 实验性 | 任意 |

## 容错

容错功能检测训练期间的挂起并自动重启工作负载。它使用基于分段的监控，为设置、训练步骤和检查点操作设置不同的超时阈值。

### 何时使用

容错适用于以下情况：

- 在不可靠的硬件上或进行大规模训练时
- 瞬态故障（网络故障、GPU 错误）很常见
- 希望无需人工干预即可自动恢复

### 稳定约束

- 需要 Slurm 和 `ft_launcher`（而非 `torchrun`）
- 必须配置并可访问检查点目录
- 使用 `nvidia-resiliency-ext` 的 RankMonitorClient
- 与 NSys 性能分析不兼容

系统支持作业内重启（在同一 Slurm 分配内）和故障时启动新作业，每种方式都有可配置的限制。

## 掉队检测

NVRx 掉队检测监控跨不同秩（rank）的 GPU 性能，并识别性能缓慢的节点。它计算相对和个体的性能分数，并且如果性能低于可配置的阈值，可以选择性地终止训练。

### 何时使用

掉队检测在以下情况下很有用：

- 进行大规模训练时，一个慢速节点会降低整体吞吐量
- 希望了解每个秩的 GPU 性能
- 需要识别持续的硬件问题

### 稳定约束

- 需要 `nvidia-resiliency-ext`
- 开销很小，但可以通过 `profiling_interval` 进行调整
- 默认**不**停止训练；必须将 `stop_if_detected` 显式设置为 `True` 才能自动终止

## 抢占处理

抢占处理在训练作业收到终止信号（默认：SIGTERM）时提供优雅的关闭。它会在退出前保存检查点，以保留训练进度。

### 何时使用

抢占处理在以下情况下很重要：

- 在具有作业时间限制的共享集群上运行时
- 更高优先级的作业可能会抢占你的分配
- 希望在作业终止时最大限度地减少工作损失

### 稳定约束

- `PreemptionPlugin` 是 Slurm 特有的
- 通过 `exit_signal_handler` 的直接配置适用于任何集群
- 信号检测发生在每个训练步骤结束时

## 异步检查点保存

异步检查点保存使用持久化的后台工作进程，将检查点 I/O 与训练计算重叠。在调度保存后，训练会立即继续，而不是阻塞直到写入完成。

### 何时使用

异步保存在以下情况下很有价值：

- 检查点保存时间占步骤时间的很大一部分时
- 正在使用 `torch_dist` 检查点格式时

### 稳定约束

- 需要 `ckpt_format="torch_dist"`
- 其他格式（zarr, fsdp_dtensor）不支持异步保存
- 必须启用持久化的检查点工作进程

## 本地检查点

本地检查点首先将检查点数据保存到节点本地存储，然后在可配置数量的节点之间进行复制。这避免了在关键路径上写入共享网络存储的延迟。

### 何时使用

本地检查点在以下情况下很有用：

- 共享存储的检查点写入是检查点间隔的瓶颈时
- 希望在不依赖网络文件系统可用性的情况下，更快地从节点故障中恢复
- 进行大规模训练时，网络存储争用很常见

### 稳定约束

- 节点本地存储必须具有至少容纳一个检查点的足够容量
- 必须配置复制度以应对预期的故障率
- 需要兼容的检查点格式（参见 [skills/resiliency/SKILL.md](../skills/resiliency/SKILL.md)）

## 重运行状态机

重运行状态机是一个实验性功能，用于将意外结果（NaN 损失、尖峰损失）归因于瞬态错误、持续的硬件故障、

或正确但意外的结果。它通过在相同和不同的 GPU 上重新运行计算来实现。

### 何时使用

在以下情况下考虑使用重新运行状态机：

- 需要自动化的 NaN 检测和归因
- 希望区分硬件故障与训练不稳定性

### 稳定约束

- Alpha 级别功能；完全集成有限
- 三种模式：`disabled`、`validate_results`、`report_determinism_stats`
- 使用特定的退出代码（16, 17）来控制作业行为

## 进程内重启

进程内重启通过在同一操作系统进程内重启训练函数，提供自动故障恢复。这避免了启动新作业、启动容器和创建新 CUDA 上下文带来的开销。

### 何时使用

进程内重启适用于以下情况：

- 软件故障（异常、死锁）比硬件故障更常见
- 重启延迟很重要，并且希望避免完整的作业重新启动
- 可以接受实验性状态和兼容性约束

### 稳定约束

- 需要 PyTorch >= 2.5.1 和 NCCL >= 2.26.2
- 与 NeMo-Run 或 Slurm 抢占插件不兼容
- 需要特定的环境变量（`NCCL_NVLS_ENABLE=0` 等）
- PyTorch NCCL 看门狗超时必须超过 `hard_timeout`
- 支持节点级别和 rank 级别的重启粒度

进程内重启不适用于硬件级别的故障，例如交换机故障或网络分区。为了实现全面的容错能力，应将其与作业级容错结合使用。

## 实际注意事项

1. 没有单一的弹性功能可以覆盖所有故障模式。推荐的方法是分层使用功能（例如，容错 + 掉队检测 + 异步检查点）。
2. 并非所有配方都默认启用弹性功能。请检查并显式启用。
3. 代码库中存在两个掉队检测器（NVRx 和旧版 MCore）。请使用 NVRx 版本；不要同时启用两者。
